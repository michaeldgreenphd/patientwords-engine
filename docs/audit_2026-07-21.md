## 1 · README accuracy (engine)

### 1.1 Verdict

The README's parameter tables, setup, model-evaluation, publishing, and auto-interp
sections verify against the code. But the document has fallen one architectural
generation behind main on three axes, and it is internally inconsistent in one place:

1. **Zero Jacobian-lens coverage.** No occurrence of "jlens", "Jacobian", or "lens"
   anywhere, despite seven landed scripts, a workflow (`jlens_readout.yml`), tests,
   and three live site figures. The method credit the code itself carries (Gurnee et
   al., Transformer Circuits 2026; reference implementation `anthropics/jacobian-lens`,
   Apache-2.0; Neuronpedia-hosted deployment) is absent. Redline R13.
2. **Two of six workflows undescribed.** `logits_evaluation.yml` (the entire
   cross-model behavioral path the site's model selector depends on) and
   `jlens_readout.yml` have no README presence; the push-to-run trigger model and the
   guarded ops layer (`fire_trigger.py`, journal, $2/day ceiling, daily Routine) are
   documented only in CLAUDE.md and `.github/trigger/README.md` — invisible to an
   external reader. One orphan mention survives at README:460 ("push
   `.github/trigger/archive-renders.json`") with no explanation of the model.
   Redlines R12, R19.
3. **Stale numbers and superseded framing.** "About 130 graphs so far" — main's
   committed batch summaries alone hold 628 hosted result rows (~1,400 graph
   generations; independently recomputed twice). The private-repo Actions-minutes
   framing predates the repos going public (2026-07-09). The "$4–6 against a $20
   budget" plan was executed and superseded by the pre-registered Tier B run
   ($8 generation ceiling, $2/day operational ceiling, ~$0.002/accepted pair on
   haiku; committed sidecars already sum to $13.00 lifetime across 55 sidecars).
   Redlines R16–R18.
4. **The 4quadrant description is wrong, not just stale.** The code renamed both
   axes (morphosyntax × lexicon), renamed both badges (Register shift Δ / Variety
   shift Δ), and re-lettered the edges: the lexicon swap is A→C / B→D and the
   morphosyntax swap is A→B / C→D (`batch_eval.py:14-29,107-125,717-722`). The
   README's mapping is **inverted** relative to shipped renders, and its own
   "Reading the 4quadrant renders" section already uses the new vocabulary — the
   document contradicts itself. Redline R4.

Also verified as *accurate* under adversarial checking, worth noting because both
had been flagged as unverifiable: the steering first-live-trial numbers

### P0 — blocks outside replication

1. **The raw measurements behind 10 of 14 published batches are not on public
   main.** For every Tier B batch in the site payload (2026-07-10 onward),
   `trace_out/` on engine main contains zero `batch_summary*.json` parts, while
   the payload publishes their measurements (748 scenarios). Trace outputs commit
   to the dispatched branch by design, so the raw record lives only on unmerged,
   deletable experiment branches; `payload.archive` is null. An outside researcher
   cannot audit any published Tier B probability back to its raw trace. *Fix:*
   merge the trace summaries (and `jlens_raw` dirs) for every published batch onto
   main, or cut a tagged raw-measurements GitHub Release with a manifest and
   record its URL in the payload and `data/provenance.json`.
2. **No committed generator for the claim-grade statistics files.** Zero engine-main
   references to `model_stats`, `convergence`, `timeline`, `retrace_consistency`,
   `drift_series`, `patch_profile`, `specialty*`, `tag_mass`, `translation_scale`,
   `jlens_swaps`, `model_provenance`. Frontend CLAUDE.md attributes
   `model_stats.json` to `paired_stats_rigor.py` — absent from main; main's
   `paired_stats.py` writes a different, simpler schema. The headline inferential
   claims (per-model penalty CIs, simultaneous CIs, BH correction) are made by code
   the public cannot see. *Fix:* land the generators on main with tests; stamp each
   output with script name + engine commit sha; fix the frontend CLAUDE.md
   attribution.
3. **Environment unpinned and unrecorded.** No lock file; caret ranges only;
   torch/transformers/accelerate — the whole CPU-logits stack, the primary Tier B
   endpoint — are not package deps at all and install unpinned in CI
   (`logits_evaluation.yml:114-117`). No run records python/torch/transformers
   versions or the engine sha. The site's variance caveat cannot even be
   attributed between service drift and dependency drift. *Fix:* constraints/lock
   file; pin versions in all six workflows; extend `logits_eval.py`'s inference
   block and the batch-summary header with versions + sha + runner image.
4. **No license anywhere that covers reuse.** The engine repo has no LICENSE file
   (and `pyproject.toml:5` still says `authors = ["Neuronpedia"]`, a fork
   leftover contradicting CITATION.cff); the frontend's MIT covers "the Software"
   with no terms for the phrase datasets, measurements, or the collaborator CSV.
   CITATION.cff invites citation of data nobody is licensed to reuse. *Fix:*
   MIT/Apache-2.0 for engine code; an explicit data-license statement (e.g.
   CC BY 4.0) covering `data/simulated/`, `data/measured/`, and the frontend
   `data/` payloads, referenced from both READMEs and the CFF.

## 3 · Python vs HTML: where computation actually lives

### 3.1 The clarification

"Engine code is in html" conflates three things:

1. **`modes/` figure renders** are self-contained HTML files, but they are engine

   exports, never edited frontend-side. Nothing computes in them at view time

## 6 · Branch strategy

### 6.1 Constraints, verified against the YAML (not the docs)

1. **Push-to-run fires on any branch** — every workflow's `on.push.paths` filter
   watches exactly one trigger file, with no `branches:` filter. A merge whose
   commit changes a trigger file relative to the target branch fires that
   workflow on the target ref (the documented merge/copy danger). Creating a
   previously-absent trigger file (jlens-readout on main) also fires.
2. **The queue is per-branch — for the $0 workflows only.** circuit-trace,
   jlens-readout, logits-eval, archive-renders all carry
   `group: <name>-${{ github.ref }}`, `cancel-in-progress: false` → one running +
   one pending per branch; a third push silently evicts. Extra branches do add
   queue slots for these four.
3. **Discovery — the two PAID workflows have no concurrency stanza at all.**
   `scenario_generation.yml` and `model_evaluation.yml` have no `concurrency:`
   block (grep verified), contra CLAUDE.md:54's "every workflow has a per-branch
   concurrency group" and `fire_trigger.py`'s docstring. On the paid path the
   journal guard is the *only* queue/spend discipline. **→ execution session:**
   add the stanzas; correct CLAUDE.md and the docstring.
4. **The guard is per-checkout and branch-blind.** The journal lives in the
   checkout (`ops/trigger_journal.jsonl`, committed per-branch); entries record
   no branch field; `inflight_max_spend` and the budget check read only this
   checkout's journal + dashboard. N firing branches would each see only their
   own in-flight spend — the $2/day ceiling effectively multiplies by N, with no
   CI backstop on precisely the paid workflows (point 3).
5. **Output topology is split by design.** Generation archives commit to main via
   a dedicated worktree regardless of dispatched ref; trace/logits/jlens/eval/
   archive outputs commit to the dispatched branch (with pull-rebase retry loops;
   jlens uses `-X theirs` after a documented 2026-07-15 chunk loss). Workflows
   read the pairs file from the dispatched ref's checkout, so every main-archived
   batch needs a copy commit onto the firing branch before tracing.
6. **The two-branch cost is already observable at N=2:** main's journal holds 3
   entries, all 2026-07-09, all unresolved; main's dashboard is 12 days stale
   with `tierb.start_utc` null — which also means `tierb_split.py` *computes an
   empty holdout set on a main checkout* (it reads start_utc from the dashboard);
   the jlens handoff doc rotted within a day; `activation-patching` sits in
   TRIGGERS with no workflow on main.

### 6.2 Options

- **A — everything on main; fire only from main.** The guard's single-checkout
  worldview becomes globally correct (one journal, one dashboard, the budget
  ceiling actually enforced); output topology collapses (no per-batch copy
  commits; one tree for the collectors); merge/copy refire risk goes to zero;
  ops state stops forking. Cross-workstream parallelism survives (four separate
  $0 groups + in-run matrices on one branch). Cost: single-workflow throughput
  caps at one running + one pending chain; experimental edits share the branch
  the nightly Routine executes from; render weight accumulates on main (existing
  archive+prune flow mitigates).
- **B — main + one rolling dev branch (today's de facto state).** Keeps one
  firing checkout (guards coherent) and isolates in-flight code, but pays the
  copy-commit tax forever, leaves main's ops surface stale by design (observed),
  and makes every eventual promotion merge a hand-restored-trigger-files
  exercise where one slip on scenario-generation.json is an unguarded paid run.
- **C — per-workstream long-lived branches.** The only option that adds slots
  for the *same* workflow — and it mechanically breaks the paid-path safety
  (per-branch journals multiply the daily ceiling; no CI backstop), forks the
  dashboard against the single-writer rule, multiplies the copy/reassembly cost
  (the collectors need all parts in one checkout), multiplies the refire
  surface, and quietly sidesteps the deliberate politeness caps on the hosted
  endpoints (jlens `max-parallel: 2`). Worst option for owner-away oversight;
  the drift already observed at N=2 scales with N.
- **D — ephemeral per-run branches.** Maximum isolation, but every merge-back
  is a trigger-restore chore; whole-file journal conflicts fail closed
  (`SystemExit` — exactly what an unsupervised Routine cannot repair); a
  forgotten merge-back orphans measured data.

### 6.3 Recommendation

**Adopt A (single live checkout = main) as the target state; do not migrate
mid-flight.** Keep B untouched until the in-flight Tier B chains on
`claude/gemma-clinical-colloquial-interp-mavx04` are harvested and every journal
entry resolved — ideally at owner return. The deciding argument is that the entire
safety stack (journal, budget and settle guards, single-writer dashboard, the
Routine's standing prompt) is architected around exactly one live checkout, and
the paid workflows have no CI concurrency backstop, so branch-splitting blinds the
only spend control precisely where it matters. Branch-splitting as a parallelism
tool buys slots only for $0 workflows, and cross-workstream parallelism already
exists on one branch. For code isolation, use short-lived **non-firing** feature
branches with one rule: trigger files change only on main; feature branches keep
`.github/trigger/` byte-identical to main — then they never fire and never refire
on merge.

### 6.4 Migration plan (→ execution session; owner sign-off)

1. **Quiesce.** Wait for a cycle with no active fires; `resolve` every remaining
   journal entry on the working branch; wait out the settle window; confirm all

2. **Merge once, trigger-safe.** `git merge --no-commit`, then
   `git checkout main -- .github/trigger/` and byte-verify all five trigger

   the merge push. Take the *branch's* journal and dashboard wholesale (they are
   the live state; main's is a strict-stale subset), then hand-resolve anything
   the merge revives.
3. **Repoint the Routines** (daily + the accelerator, which must first be located
   and documented) to check out and fire from main; update
   `routine_standing_prompt.md` §1 wording and drop the §2 cross-branch copy
   step; note the prompt's own rule that changing its text requires recreating

4. **Hygiene, same session:** (a) add per-branch concurrency stanzas to
   `scenario_generation.yml` and `model_evaluation.yml`; (b) promote
