# Context pack — second-eyes review of a new Claude Code prompt

You are reviewing a draft prompt before it is given to Claude (model: Fable 5)
in a Claude Code session with full repo access. Your job: find ambiguities,
missing constraints, and scope creep in the draft; strengthen it against the
context below. The owner will paste their draft after this document.

## The project in brief

Two GitHub repos, one study:

- **patientwords-engine** (public): measures how small language models shift
  next-token predictions between clinical terminology and colloquial patient
  phrasing. Sentence pairs are generated by Claude via API, traced as
  attribution graphs through Neuronpedia's hosted circuit-tracer (gemma-2-2b),
  and measured behaviorally by CPU inference (three more models). Probabilities
  only — nothing generates or evaluates medical advice.
- **patientwords** (public): a GitHub Pages gallery presenting the results.
  Self-contained HTML pages (inline CSS/JS, no build system, no CDN). Deploys
  from `main`.

Established findings (all owner-released 2026-07-09): mean penalty −0.070
[−0.098, −0.043] on 132 pairs, replicated across 4 models; prediction flips
skew 109 care-urgency downgrades vs 12 upgrades under an owner-reviewed tier
vocabulary; the responsible feature circuit is named and causally steerable
(placebo-controlled, dose–response measured); LLM translation back to clinical
wording recovers 8/20 downgrades.

## Operating model — the prompt must not fight these

- **Nothing paid or networked runs locally.** API keys exist only as GitHub
  Actions secrets. All generation/tracing/inference runs through push-to-run
  CI: each workflow fires when its file under `.github/trigger/` changes on a
  pushed branch.
- **Queue discipline:** every workflow has a per-branch concurrency group,
  one running + one pending; pushing a third trigger change silently evicts
  the pending run. Merges that touch trigger files re-fire workflows and
  double-spend.
- **Costs:** only Claude generation/evaluation spends money (~$0.002/pair on
  haiku). Tracing and CPU inference are $0. Every paid run writes a
  `.report.json` cost sidecar. Hosted tracing throughput: 8–12 pairs/hour.
- **Data rules:** medical vocabulary lives only in JSON data files, never in
  Python source. Generated batches are append-only archives. Intentional
  misspellings in stimuli are never corrected.
- **Site rules:** every number on a page must trace to a data file in the
  repo; provenance labels stay accurate; pages are standalone (duplication of
  CSS/JS across pages is deliberate); fixed semantic palette; accessibility
  floor (skip links, focus-visible, aria-sort, etc.).
- Both repos have a `CLAUDE.md` encoding these rules; new standing rules are
  added there, not stated in chat.

## What a strong version of the new prompt contains

1. **Goal in one sentence** (mission-control dashboard + daily Routine +
   process-as-code scripts), and explicitly what is out of scope (B/C above;
   the Tier B science itself, which is pre-registered and running).
2. **The dashboard's data contract**: which JSON file(s) it reads, who
   writes them and when (the daily Routine session updates them after each
   harvest), and what panels v1 must show — queue/runs state, spend vs both
   ceilings ($8 gen + $2/day), Tier B progress vs the 1,600 target,
   findings delta, decisions waiting on the owner.
3. **The Routine's standing prompt text** as a deliverable, not an
   afterthought — it must: locate state from the repos alone, obey queue
   discipline, respect the ceilings, update dashboard data + ledger, send
   one digest push per day, and never touch trigger files except to chain
   the pre-registered Tier B batches.
4. **Boundaries stated as hard rules**: spend ceilings, what needs owner
   sign-off vs standing approval, never rewrite landed batches, never merge
   trigger files, site hard rules.
5. **Acceptance criteria**: scripts have regression tests and pass the
   existing offline suite; the dashboard is verified in a real browser with
   no console errors and renders correctly with missing/pending data; the
   Routine is created and its first firing observed (or dry-run) before the
   owner leaves Friday.
6. **Deliverable format/destination named** for each artifact: committed
   file paths in the engine repo, and what arrives on the owner's phone.

## What Fable is good at (so the prompt can lean on it)

Long-horizon autonomous execution under explicit ceilings (give authority +
budget + definition of done, not step-by-step); whole-corpus audits ("every
number traces to a source"); statistical red-teaming and pre-registration
discipline; zero-dependency front-end with an existing design system
(specify the interaction contract and data source, not the aesthetics);
encoding process rules as tested scripts; browser-verifying its own UI work.
Weak spots to guard in the prompt: it will act autonomously within stated
boundaries — so state them; phone artifacts are snapshots — say when they
must be regenerated; fresh sessions know nothing — the prompt must point at
where state lives.

## Questions to pressure-test the draft against

- Does the Routine prompt survive a session that starts with zero context?

- Are both spend ceilings stated with enforcement mechanics, not vibes?
- Is "daily digest" specified — content, length, and the only-one-per-day
  rule — so pushes stay signal, not noise?
- Does anything in the draft touch `.github/trigger/` outside the
  pre-registered Tier B chain? It shouldn't.
- Is there a Friday-morning checkpoint where the owner sees the dashboard
  and the first digest before leaving?
